# Проектирование производительности: кеширование и rate limiting

## 1. Анализ производительности

### 1.1 Часто выполняемые операции

| Endpoint | Частота | Описание |
|----------|---------|----------|
| `GET /products` | Очень высокая | Каталог товаров — главная страница магазина |
| `GET /products/{id}` | Высокая | Карточка товара — просмотр перед покупкой |
| `GET /products?tags=...` | Высокая | Фильтрация по тегам — навигация по каталогу |
| `GET /cart` | Высокая | Просмотр корзины — многократно за сессию |
| `POST /auth/login` | Средняя | Аутентификация пользователей |

### 1.2 Медленные операции

| Операция | Причина медлительности | Текущее время |
|----------|----------------------|---------------|
| `GET /products?tags=...` | Dual-read: MongoDB (фильтр по тегам) + PostgreSQL (JOIN по ID) | ~5-50 мс |
| `GET /products/{id}?extended=true` | Dual-read: PostgreSQL + MongoDB | ~3-10 мс |
| `GET /products` | PostgreSQL SELECT + COUNT | ~2-5 мс |
| `GET /cart` | JOIN cart_items + products + SUM | ~2-10 мс |
| `GET /users/search?q=...` | ILIKE с GIN-индексом | ~1-5 мс |

### 1.3 Требования к производительности

| Метрика | Целевое значение |
|---------|-----------------|
| Время отклика (p50) | < 50 мс |
| Время отклика (p99) | < 200 мс |
| Пропускная способность | > 1000 RPS на endpoint |
| Cache hit rate | > 80% для каталога |
| Доступность | 99.9% |

## 2. Стратегия кеширования

### 2.1 Кешируемые данные

| Данные | Частота чтения | Частота изменения | Обоснование кеширования |
|--------|---------------|-------------------|------------------------|
| Список товаров (пагинация) | Очень высокая | Низкая | Каталог меняется редко, читается постоянно |
| Товар по ID | Высокая | Низкая | Карточка товара стабильна |
| Фильтрация по тегам | Высокая | Низкая | Результаты стабильны между изменениями каталога |
| Корзина пользователя | Средняя | Средняя | Корзина читается чаще, чем изменяется |

### 2.2 Стратегии кеширования

#### Cache-Aside — для всех endpoints

Выбрана стратегия **Cache-Aside**, потому что:
- Приложение контролирует, что попадает в кеш
- Кеш заполняется только реально запрашиваемыми данными (нет wasteful preloading)
- При сбое Redis приложение продолжает работать напрямую с БД
- Простая реализация и отладка

Алгоритм:
```
1. Проверить ключ в Redis
2. Cache HIT → вернуть из Redis (десериализация JSON)
3. Cache MISS → запросить БД → сохранить в Redis с TTL → вернуть
```

Альтернативы и причины отказа:
- **Read-Through**: требует отдельного cache-слоя, избыточно для текущего масштаба
- **Write-Through**: увеличивает latency записи, не оправдано при низкой частоте записей
- **Write-Back**: риск потери данных при сбое Redis, неприемлемо для e-commerce

### 2.3 TTL

| Ключ | TTL | Обоснование |
|------|-----|-------------|
| `products:list:{skip}:{limit}` | 60 сек | Каталог обновляется нечасто, допустима задержка до 1 мин |
| `products:{id}` | 120 сек | Карточка товара стабильна, допустима задержка до 2 мин |
| `products:{id}:extended` | 120 сек | Расширенные атрибуты из MongoDB меняются редко |
| `products:tags:{sorted_tags}` | 60 сек | Аналогично списку товаров |
| `cart:{user_id}` | 30 сек | Корзина персональна и может меняться чаще |

### 2.4 Стратегия инвалидации кеша

Используется **явная инвалидация** при мутирующих операциях:

| Мутация | Инвалидируемые ключи |
|---------|---------------------|
| `POST /products` (создание) | `products:list:*`, `products:tags:*` |
| `DELETE /products/{id}` | `products:{id}*`, `products:list:*`, `products:tags:*` |
| `PUT /products/{id}/attrs` | `products:{id}*`, `products:list:*`, `products:tags:*` |
| `POST /cart/items` | `cart:{user_id}` |

Паттерн инвалидации — **scan + delete**: поиск ключей по шаблону через `SCAN` и удаление пакетом. Подходит для текущего масштаба. При масштабировании можно перейти на Redis pub/sub или теговую инвалидацию.

## 3. Проектирование rate limiting

### 3.1 Endpoints, требующие rate limiting

| Endpoint | Уровень | Причина |
|----------|---------|---------|
| `POST /auth/register` | Nginx (по IP) | Защита от массовой регистрации ботов |
| `POST /auth/login` | Nginx (по IP) | Защита от brute-force атак на пароли |
| `GET /users/search` | Nginx (по IP) | Предотвращение scraping данных пользователей |
| `POST /cart/items` | Приложение (по user_id) | Защита от спама в корзину |
| Все endpoints | Nginx (по IP) | Общая защита от DDoS |

### 3.2 Алгоритмы rate limiting

#### Nginx: Встроенный `limit_req`

- **Алгоритм**: Leaky Bucket — запросы обрабатываются с фиксированной скоростью, избыточные отклоняются
- **Обоснование**: встроен в Nginx, нулевая задержка, работает на уровне gateway до микросервисов
- **Конфигурация**:
  - `auth_zone`: 10 req/s, burst=5 — жёсткий лимит для аутентификации
  - `api_zone`: 50 req/s, burst=20 — общий лимит для API
- **Режим**: `nodelay` — при превышении burst сразу 429, без буферизации

#### Приложение: Fixed Window Counter (через Redis)

- **Алгоритм**: Fixed Window Counter — подсчёт запросов в фиксированном окне (1 минута)
- **Обоснование**: простая реализация через Redis `INCR` + `EXPIRE`, атомарные операции
- **Конфигурация**:
  - Ключ: `ratelimit:cart:{user_id}:{window_number}`
  - Лимит: 30 запросов в минуту на пользователя
  - TTL ключа: 60 секунд (автоматическая очистка)

Альтернативы:
- **Token Bucket**: сложнее в реализации (нужен Lua-скрипт), но обеспечивает burst
- **Sliding Window Log**: высокое потребление памяти (хранит timestamp каждого запроса)
- **Sliding Window Counter**: точнее Fixed Window, но сложнее — оправдан при масштабировании

### 3.3 Лимиты

| Endpoint | Лимит | Окно | Уровень | Идентификатор |
|----------|-------|------|---------|---------------|
| `POST /auth/*` | 10 req/s | — | Nginx | IP-адрес |
| Все API endpoints | 50 req/s | — | Nginx | IP-адрес |
| `POST /cart/items` | 30 req | 1 мин | Приложение | user_id |

### 3.4 Ответ при превышении лимита

HTTP-ответ при превышении:
```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
X-RateLimit-Limit: 30
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1715100120
Retry-After: 45

{"detail": "Too Many Requests"}
```

Заголовки:
- `X-RateLimit-Limit` — максимальное количество запросов
- `X-RateLimit-Remaining` — оставшееся количество запросов
- `X-RateLimit-Reset` — Unix timestamp сброса окна
- `Retry-After` — секунды до сброса (только при 429)

## 4. Анализ влияния на производительность

### 4.1 Влияние кеширования

| Метрика | Без кеша | С кешем (cache hit) | Улучшение |
|---------|----------|---------------------|-----------|
| `GET /products` latency | 2-5 мс (PG) | < 1 мс (Redis) | 2-5x |
| `GET /products/{id}?extended=true` latency | 3-10 мс (PG + Mongo) | < 1 мс (Redis) | 3-10x |
| `GET /products?tags=...` latency | 5-50 мс (Mongo + PG) | < 1 мс (Redis) | 5-50x |
| `GET /cart` latency | 2-10 мс (PG JOIN) | < 1 мс (Redis) | 2-10x |
| Нагрузка на PostgreSQL | 100% запросов | ~20% (при hit rate 80%) | 5x снижение |
| Нагрузка на MongoDB | 100% запросов | ~20% (при hit rate 80%) | 5x снижение |

### 4.2 Влияние rate limiting

- Защита от DDoS: отсечение избыточных запросов на уровне Nginx (до микросервисов)
- Защита от brute-force: ограничение попыток аутентификации
- Справедливость: один пользователь не может монополизировать ресурсы
- Предсказуемость: гарантированное время отклика для легитимных пользователей

### 4.3 Метрики для мониторинга

| Метрика | Описание | Источник | Целевое значение |
|---------|----------|----------|-----------------|
| Cache hit rate | Доля запросов из кеша | Redis INFO stats | > 80% |
| Cache miss rate | Доля промахов | Redis INFO stats | < 20% |
| Cache eviction rate | Скорость вытеснения ключей | Redis INFO stats | ~ 0 |
| Redis memory usage | Потребление памяти | Redis INFO memory | < 100 МБ |
| Redis latency | Время ответа Redis | Redis LATENCY | < 1 мс |
| Rate limit 429 count | Количество отклонённых запросов | Nginx access log | < 1% от всех |
| API p50/p95/p99 latency | Время ответа API | Application metrics | p99 < 200 мс |
| DB query count/sec | Запросов к БД в секунду | PostgreSQL pg_stat | Снижение после кеша |

### 4.4 Измерение эффективности кеширования

Cache hit rate рассчитывается по формуле:

```
hit_rate = keyspace_hits / (keyspace_hits + keyspace_misses) * 100%
```

Получение метрик:
```bash
redis-cli INFO stats | grep keyspace
```

Мониторинг по ключам:
```bash
redis-cli MONITOR  # реальное время (для отладки)
redis-cli INFO keyspace  # количество ключей по базам
redis-cli DBSIZE  # общее количество ключей
```
